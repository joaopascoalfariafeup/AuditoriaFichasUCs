Apreciação da Ficha da Unidade Curricular

Aspeto a Avaliar Apreciação Qualitativa Apreciação Quantitativa (0–3) (a)
Objetivos/resultados de aprendizagem Objetivos bem estruturados e detalhados, com verbos de ação adequados e alinhamento claro com os níveis cognitivos apropriados para a UC 3
Tópicos programáticos Programa conciso mas abrangente, cobrindo adequadamente os tópicos essenciais de bases de dados relacionais 2
Metodologias de ensino e aprendizagem Metodologias apropriadas combinando teoria e prática, mas descrição pouco detalhada das atividades específicas 2
Metodologias de avaliação Estrutura de avaliação adequada mas com algumas limitações na flexibilidade e detalhamento das componentes 2
Apreciação global Ficha bem estruturada com objetivos claros, mas com oportunidades de melhoria na especificação de alguns elementos 2.25

(a) Escala: 0 - Ausente ou totalmente inadequado; 1 - Parcialmente adequado; 2 - Largamente adequado; 3 - Totalmente adequado.

Pontos fortes

  • Objetivos de aprendizagem muito bem formulados, específicos e mensuráveis, com verbos de ação adequados aos níveis cognitivos da taxonomia de Bloom (explicar, criar, mapear, identificar, etc.)
  • Cobertura abrangente e equilibrada dos tópicos fundamentais de bases de dados relacionais, desde modelação conceptual até otimização e sistemas NoSQL
  • Combinação adequada de componentes teóricas e práticas, com uso de ferramentas apropriadas (SQLite)
  • Estrutura de avaliação que combina projeto prático (25%) com exame final (75%), permitindo avaliar competências práticas e teóricas
  • Definição clara das condições de frequência e regras de melhoria de classificação
  • Coerência entre as versões portuguesa e inglesa da ficha

Pontos fracos

  • Tópicos programáticos apresentados de forma muito concisa, sem mapeamento explícito para os objetivos de aprendizagem
  • Metodologias de ensino descritas de forma genérica, sem detalhar atividades específicas ou estratégias pedagógicas diferenciadas
  • Ausência de mapeamento claro entre componentes de avaliação e objetivos de aprendizagem específicos
  • Projeto não sujeito a reavaliação, limitando oportunidades de recuperação para estudantes
  • Falta de especificação sobre a natureza e âmbito do projeto de grupo

Sugestões de melhoria

  • Expandir a secção "Programa" para incluir mapeamento explícito entre tópicos e objetivos, por exemplo: "T1. Sistemas de gestão de bases de dados: características principais e arquitetura (O1); T2. Modelação conceptual: diagramas de classes UML para bases de dados (O2); T3. Modelo relacional: mapeamento de modelos conceptuais para esquemas relacionais (O3)..."
  • Detalhar as metodologias de ensino especificando: "As aulas teóricas incluem exposição dos conceitos, análise de casos de estudo e discussão de exemplos práticos. As aulas práticas envolvem resolução de exercícios de modelação, implementação de bases de dados em SQLite, e desenvolvimento progressivo do projeto de grupo com acompanhamento docente."
  • Especificar o mapeamento das componentes de avaliação para os objetivos: "Projeto (25%): avalia objetivos O2, O3, O6, O7, O9 através da modelação, implementação e consulta de uma base de dados; Exame final (75%): avalia todos os objetivos O1-O12 através de questões teóricas e práticas"
  • Considerar permitir reavaliação do projeto ou componente alternativa para estudantes em situações excecionais, alterando: "O projeto pode ser reavaliado em casos excecionais devidamente justificados, mediante aprovação do docente responsável"
  • Acrescentar descrição do projeto: "O projeto consiste no desenvolvimento completo de uma base de dados para um domínio específico, incluindo modelação conceptual, normalização, implementação em SQL e desenvolvimento de consultas complexas, realizado em grupos de 3-4 estudantes"
